# specs/ — per-project migration plans

Spec files are **generated by the agent**, not shipped in the template. Each file is **one unit of work** and is created by `/plan-migration` (Phase 1) after analyzing the source; tasks inside are implemented one-per-loop by `/implement-next` (Phase 2).

## Naming (descriptive — the folder is the history of what was built)
`specs/<NN>-<type>-<slug>.md`, increasing `NN`. Examples:
- `specs/01-initial-migration-acme.md` — full from-scratch migration
- `specs/02-fix-footer-mobile.md` — a later fix
- `specs/03-redesign-hero.md` — a redesign of one area

## Rules
- **Every implementable line is a checkbox (`- [ ]`).** Tasks, sub-steps, tests, asset actions, QA items.
- Structure follows [`../docs/spec-template.md`](../docs/spec-template.md): Header · Purpose · Phases (foundation→sections→pages/nav/i18n/SEO→QA) · Assets register · Decision log · Validation/QA.
- **Tests are planned here as checkbox items** (named `tests/*.test.js`) and implemented during the loop — never pre-created.
- A spec file is the single source of truth for its unit of work; the agent ticks items as done and never deletes the file (it's the record).
